# Installable PWA for octopus — design

**Date:** 2026-07-02
**Status:** Approved for planning

## Goal

Make octopus installable as a Progressive Web App: it adds to the phone home
screen, launches **standalone** (no browser chrome), carries the terminal theme
into the OS status bar, and loads instantly on repeat visits via an app-shell
precache. Data continues to come from the backend — no offline data, no sync.

This builds on the just-completed mobile-responsive work; it is the packaging
layer that makes the responsive UI feel like a native app.

## Non-goals (YAGNI)

- No offline **data** (no API-response caching, no write queue, no sync).
- No push notifications.
- No custom install-prompt UI — the browser's native install affordance is enough.
- No desktop-install polish beyond what the manifest provides for free.

## Approach

Add **`vite-plugin-pwa`** (dev dependency) in **`generateSW`** mode. At build it
emits, into the existing `../static` output (`frontend/vite.config.js` `outDir`):

- `manifest.webmanifest`
- a Workbox service worker (`sw.js`) whose precache list is Vite's hashed
  JS/CSS/font assets (the plugin injects the revisioned file list — this is why
  we use the plugin rather than a hand-written SW that can't track hashed names)
- the SW registration glue (auto-injected into the app entry)

`registerType: 'autoUpdate'` — a new deploy silently activates on the next load.
Appropriate for a single-user tool; no update-prompt UI needed.

No application/runtime code changes beyond `vite.config.js`, `index.html`, and
new icon assets.

## Components

### 1. Icons — `frontend/public/`

Motif: the app's signature `> ▋` prompt (mint `#3ef5c4` glyph on near-black
`#06070a`). Authored as **one source SVG**, rasterized to PNGs via a **small
headless-chromium script** (chromium is already used for screenshot verification
in this repo — no second dependency). Icons are committed as static brand assets
(generated once, not per build).

Deliverables:
- `icon-192.png` (192×192, `purpose: any`)
- `icon-512.png` (512×512, `purpose: any`)
- `icon-512-maskable.png` (512×512, glyph inset into the maskable **safe zone** —
  ~80% central area — so Android's circle/squircle mask never clips it;
  full-bleed near-black background)
- `apple-touch-icon.png` (180×180, **no transparency** — iOS composites on white
  otherwise; use the near-black background)
- `favicon.svg` and `favicon.ico` (browser tab / bookmark)

Candidates will be rendered for user approval before being wired into the
manifest.

### 2. Manifest — generated by the plugin

- `name: "Octopus"`, `short_name: "Octopus"`
- `display: "standalone"`
- `start_url: "/"` (hash router resolves the route client-side)
- `scope: "/"`
- `theme_color: "#06070a"` (near-black canvas — status bar blends with the shell)
- `background_color: "#06070a"` (launch splash background)
- `icons`: the 192 and 512 `any` icons plus the 512 `maskable` icon
- `description: "Self-hosted hub for your freelance business — projects, tasks, contacts, calendar, notes."`

### 3. `index.html` meta (currently has none of these)

- `<meta name="theme-color" content="#06070a">`
- `<meta name="apple-mobile-web-app-capable" content="yes">`
- `<meta name="apple-mobile-web-app-status-bar-style" content="black-translucent">`
- `<link rel="apple-touch-icon" href="/apple-touch-icon.png">`
- `<link rel="icon" href="/favicon.svg" type="image/svg+xml">` (+ `favicon.ico` fallback)
- `<meta name="description" content="Self-hosted hub for your freelance business — projects, tasks, contacts, calendar, notes.">`
- Keep the existing `<title>Octopus</title>` and viewport meta.

### 4. Service worker scope

Precache the **app shell only**. The SW must **never** cache or intercept
`/api/*` (dynamic, authenticated):

- `navigateFallback: '/index.html'` — SPA navigations resolve to the shell.
- `navigateFallbackDenylist: [/^\/api\//]` — API requests bypass the SW navigate
  fallback entirely.
- No `runtimeCaching` for the API. (Precache covers only the static build assets.)

Offline behavior: the shell loads and renders the terminal frame; data fetches
fail exactly as they do today, surfacing the existing `[ ERR ]` banners. No new
error-handling code.

### 5. Backend serving (integration risk)

The Rust static handler (Phase-1A "static serving" task) serves the SPA with a
catch-all → `index.html` fallback. For the PWA to work:

- `sw.js`, `manifest.webmanifest`, and the icon files must be served as **real
  files with correct content-types** (`sw.js` → `application/javascript`;
  `manifest.webmanifest` → `application/manifest+json`), and **must not** be
  swallowed by the SPA `index.html` fallback.
- The SW must be served at **root scope** so it can control the whole app.

The plan must **verify** the existing static handler matches real files before
falling back (it should, since it serves the hashed `assets/` today), and add
explicit handling only if a gap is found. This is the one place a backend change
might be needed; the default expectation is none.

## Data flow

No data-flow or API changes. Pure packaging/presentation. The SW sits between the
browser and static assets only; all `/api/*` traffic is untouched.

## Error handling

Unchanged. Offline or failed fetches use the existing `[ ERR ]` banners. The SW
adds no new user-facing error surface.

## Testing / verification

- **Build:** `npm run build` (from `frontend/`) emits `manifest.webmanifest`,
  `sw.js`, and the icon files into `../static/`, and injects SW registration.
  Existing `npx vitest run` stays green (no logic touched).
- **Automated (CDP harness — backend :8090, vite :5174, chromium :9222):**
  - `Page.getAppManifest` returns the parsed manifest with **no installability
    errors** (valid icons incl. a maskable ≥512, `start_url`, `display`,
    `theme_color`).
  - `navigator.serviceWorker.getRegistration()` resolves to an active
    registration after load.
  - `GET /manifest.webmanifest`, `/sw.js`, `/icon-192.png` each return **200**
    with the correct content-type (assert **not** `text/html` — i.e. not the SPA
    fallback). Test against the **built app served by the Rust backend**, since
    that is what exercises the static-serving integration (the vite dev server
    serves these differently).
- **Visual:** a 375px screenshot confirming `theme-color` is applied and the
  icon renders correctly; inspect the rendered icon candidates before finalizing.
- **Manual (note in plan):** true "add to home screen" + standalone launch can
  only be fully confirmed on a real device; the automated manifest/SW/content-type
  checks are the acceptance gate here.

## Build order (one spec)

1. **Icons** — source SVG + chromium rasterization script → commit the PNG/ICO/SVG
   assets to `frontend/public/`. (Independently reviewable: the rendered icons.)
2. **PWA wiring** — add `vite-plugin-pwa`, configure `generateSW` (manifest,
   theme/background colors, icons, `navigateFallback` + `denylist`,
   `registerType: 'autoUpdate'`), add the `index.html` meta/link tags. Build and
   run the automated manifest/SW checks.
3. **Backend serving verification** — confirm `sw.js`/`manifest.webmanifest`/icons
   are served with correct content-types and not swallowed by the SPA fallback;
   fix the static handler only if a gap is found.

## Out of scope (restated)

No offline data, no push, no custom install UI, no API caching, no backend
feature changes beyond (if needed) static content-type/fallback correctness.
